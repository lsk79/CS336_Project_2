import psycopg2
import sys
from getpass import getpass

def main():
    if len(sys.argv) < 2:
        print("Usage: python script.py \"SELECT * FROM table;\"")
        sys.exit(1)

    query = sys.argv[1]

    # Secure credential input
    db_name = input("Database name: ")
    user = input("Username: ")
    password = getpass("Password: ")

    try:
        conn = psycopg2.connect(
            dbname=db_name,
            user=user,
            password=password,
        )

        cur = conn.cursor()
        cur.execute(query)
        conn.commit()
        cur.close()
        conn.close()

    except psycopg2.Error as e:
        print("Database error:", e)

if __name__ == "__main__":
    main()

Input: "When creating file that contain the subset of a large SQL script that has all the CREATE TABLES how do I know which INSERT statement to choose in order to insure that all the right data are there"

Output: "To know exactly which INSERT statements to choose, you have to satisfy the database's Referential Integrity—specifically, its Foreign Key Constraints. If Table B relies on Table A, you cannot insert a row into Table B unless the exact matching row in Table A is also included in your file.To ensure all the "right" data is there, you have to follow the Dependency Tree of your database. There are two main ways to figure out which rows to pick:

Method 1: The "Bottom-Up" Approach (Best for testing): Instead of starting with the setup tables, you start with the most important "transaction" or "event" table in your database and work backward to find its parents.Pick your core data: Let's say your main table is Orders. You select 10 random INSERT statements for the Orders table. Identify the parents: Look at those 10 Orders rows. What customer_ids do they use? What product_ids do they use?
Fetch the required parents: Go to your Customers table and pick only the INSERT statements for the specific customer_ids you found in step 2. Go to the Products table and do the same. Repeat: If the Products table has a category_id, look at the categories used by your selected products, and go fetch those specific rows from the Categories table. This guarantees that every single foreign key is satisfied and you have a perfect, unbroken slice of reality.

Method 2: The "Top-Down" Approach: If you want to build the data from the ground up, you go in the exact opposite direction. Insert all independent Lookup Tables first: Find tables that have NO foreign keys (e.g., States, Statuses, Categories). You usually keep 100% of the inserts for these because they are small and don't depend on anything. Pick random Parents: Go to your Users or Customers table. Pick 10 random inserts. Note their IDs. Filter the Children: Go to the Orders table. Look for inserts where the customer_id matches one of the 10 you picked in step 2. Delete all other inserts."

Input: "“What steps should I follow to extract a SQL query from a text response?”
Ouput: "When you extract a SQL query from an LLM response, you are essentially building a "funnel." You start with a big, messy block of text and pass it through a series of filters until only the pure, safe SQL remains at the bottom.

Step 1: Target the Markdown (The "Happy Path")
Most modern LLMs format their code inside Markdown blocks. Your first step should always be to use Regular Expressions (Regex) to search for these blocks.
Action: Search for text sandwiched between ```sql and ``` (or just ``` and ```).
Why: This instantly drops 90% of the conversational text like "Here is your query:" or "Hope this helps!"
Code Concept: `re.search(r'

### Step 2: Establish a Fallback (The "Lazy LLM" Path)
Sometimes, if you prompt an LLM very strictly (e.g., "Output ONLY valid SQL"), it will actually listen to you and skip the Markdown blocks entirely. If your Step 1 regex fails, you need a backup plan.
* **Action:** If no markdown block is found, treat the *entire* raw text response as your starting string.
* **Why:** If you don't have a fallback, your code will crash every time the LLM formats its answer as plain text.

### Step 3: Strip Stray Artifacts
Whether you got your string from Step 1 or Step 2, it probably still has invisible junk attached to it.
* **Action:** Remove leading and trailing whitespaces, newlines, stray backticks (`), and stray quotation marks.
* **Why:** A query that looks like `  SELECT * FROM users;  ` or `` `SELECT * FROM users;` `` will often cause syntax errors depending on the database driver you are using.
* **Code Concept:** `raw_sql.strip().strip('`').strip('"')`

### Step 4: Handle Comments and Semicolons
LLMs love to add explanations *inside* the SQL query using comments. Sometimes they also add multiple statements separated by semicolons.
* **Action:** (Optional but recommended) Remove inline comments (lines starting with `--`). If your database driver only accepts one statement at a time, split the string by `;` and keep only the first valid part.
* **Why:** Drivers like Python's `sqlite3` or `psycopg2` sometimes throw errors if you pass them multiple statements at once or if trailing comments mess up the execution formatting.

### Step 5: Validate the Command (The Security Check)
Before you ever send the extracted string to your database, you must verify that it actually looks like SQL and is safe to run.
* **Action:** Check the very first word of your cleaned string. Ensure it is in an approved list (e.g., `SELECT`, `WITH`). If you are only doing data retrieval, explicitly reject the string if it starts with `DROP`, `DELETE`, `UPDATE`, or `INSERT`.
* **Why:** This prevents your app from accidentally trying to execute the sentence "Sorry, I don't know the answer" as a database command, and it protects your database from accidental deletion if the LLM hallucinates a `DROP TABLE` command.
* **Code Concept:** `if not clean_sql.upper().startswith(('SELECT', 'WITH')): raise Error()`

### Summary Pipeline
`Raw Text` ➔ `Regex Extraction` ➔ `Fallback Check` ➔ `Whitespace Strip` ➔ `Comment Cleanup` ➔ `Keyword Validation` ➔ **`Clean SQL`**

Input: "How can I use Python Paramiko to SSH into a remote Linux server, securely ask the user for their username and password with getpass, run a remote Python script with a SQL query as a command-line argument, and capture the output back into my local Python program?"

Output: "Use Paramiko for the SSH part of the local controller program. The basic structure is:

local Python program
-> ask for SSH username/password securely
-> build a safe remote command
-> SSH into iLab
-> run python3 ilab_script.py with the SQL query
-> capture stdout and stderr
-> print the result locally

A basic version is:

import paramiko
import getpass
import shlex


def run_remote_sql_query(sql_query: str) -> tuple[int, str, str]:
    hostname = "ilab.cs.rutgers.edu"
    username = input("Enter your iLab/Rutgers NetID: ")
    password = getpass.getpass("Enter your iLab/Rutgers password: ")

    remote_script_path = "~/ilab_script.py"
    remote_command = f"python3 {remote_script_path} {shlex.quote(sql_query)}"

    client = paramiko.SSHClient()
    client.load_system_host_keys()
    client.set_missing_host_key_policy(paramiko.AutoAddPolicy())

    try:
        client.connect(
            hostname=hostname,
            username=username,
            password=password,
            look_for_keys=False,
            allow_agent=False,
            timeout=15
        )

        stdin, stdout, stderr = client.exec_command(remote_command)

        output = stdout.read().decode("utf-8", errors="replace")
        error_output = stderr.read().decode("utf-8", errors="replace")
        exit_code = stdout.channel.recv_exit_status()

        return exit_code, output, error_output

    finally:
        client.close()


if __name__ == "__main__":
    sql = "SELECT * FROM application LIMIT 5;"
    exit_code, output, error_output = run_remote_sql_query(sql)

    print("--- Remote Output ---")
    print(output)

    if exit_code != 0:
        print("--- Remote Error ---")
        print(error_output)

    print("Exit code:", exit_code)

Input: "So, if I need to pass a SQL SELECT query as an argument to a remote Python script over SSH, how should I safely quote or escape the query so spaces, semicolons, and quotation marks do not break the shell command?"

Output: "Use Python's shlex.quote() on the SQL query before inserting it into the remote shell command.

The recommended pattern is:

import shlex

sql_query = "SELECT * FROM application WHERE applicant_income_000s > 100 LIMIT 5;"
remote_script_path = "~/ilab_script.py"
remote_command = f"python3 {remote_script_path} {shlex.quote(sql_query)}"

This safely turns the SQL query into one shell-safe argument. Even if the SQL query contains spaces, semicolons, single quotes, or double quotes, the remote shell will still pass it as one argument to the Python script.

Without quoting, this command can break:

python3 ilab_script.py SELECT * FROM application WHERE name = 'John Smith';

The shell would treat SELECT, *, FROM, application, WHERE, and the rest as separate arguments. But the remote script expects the whole query to be sys.argv[1].

With shlex.quote(), the command becomes shell-safe, even if it looks ugly. That is okay because the shell interprets it correctly.

A useful helper function is:

import shlex


def build_remote_sql_command(sql_query: str, remote_script_path: str) -> str:
    quoted_query = shlex.quote(sql_query)
    return f"python3 {remote_script_path} {quoted_query}"

If the script path might contain spaces, quote the script path too:

remote_command = f"python3 {shlex.quote(remote_script_path)} {shlex.quote(sql_query)}"

For the project, the key line is:

command = f"python3 {remote_script_path} {shlex.quote(sql_query)}"

That is what protects the SQL query from breaking the SSH command."
